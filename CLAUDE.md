# CLAUDE.md 开发准则

## 概览

本文件用于指导在当前仓库内进行的全部开发与文档工作，确保输出遵循强制性标准并保持可审计性。
本项目使用 Model Context Protocol（MCP）提供的多种工具服务（详见下文“🔌 MCP 集成与工具调用链总览”）。

**上下文信息要求**

- 在编码前至少分析 3 个现有实现或模式，识别可复用的接口与约束。
- 绘制依赖与集成点，确认输入输出协议、配置与环境需求。
- 弄清现有测试框架、命名约定和格式化规则，确保输出与代码库保持一致。
- **优先使用 context7 查询编程库文档**，避免过度依赖网页搜索或猜测。
- **使用 github.search_code 搜索开源实现示例**，学习最佳实践。
- **使用 desktop-commander 进行本地文件分析和数据处理**，绝对优先于 bash 命令。

---

## 语言使用强制规范

- ⚠️ **绝对强制使用简体中文**：所有 AI 回复、文档、注释、日志、提交信息等一切可使用任意语言的内容，必须强制使用简体中文。
- 唯一例外：代码标识符（变量名、函数名、类名等）遵循项目既有命名约定。
- 违反此规范的任何输出必须立即重写为简体中文。

**唯一例外**：代码标识符（变量名、函数名、类名、包名等）遵循项目既有命名约定（通常使用英文）。

---

## 🔒 强制验证机制

- 必须拒绝一切 CI、远程流水线或人工外包验证，所有验证均由本地 AI 自动执行。
- 每次改动必须提供可重复的本地验证步骤（脚本、指令或推理说明），并在失败时立即终止提交。
- 验证过程中如遇工具缺失或测试覆盖不足，必须在任务文档中记录原因和补偿计划。
- 若存在无法验证的部分，必须先补足验证能力或将任务退回，不允许带缺陷交付。

## 🤝 质量审查规范

**审查职责（Claude Code 独立执行）：**

- 使用 sequential-thinking 进行深度审查分析（通过 MCP `sequential-thinking` 服务触发，详见后文 MCP 章节）。
- 输出技术维度评分（代码质量、测试覆盖、规范遵循）。
- 输出战略维度评分（需求匹配、架构一致、风险评估）。
- 输出综合评分（0-100）和明确建议（通过/退回/需讨论）。
- 生成 `.claude/verification-report.md` 审查报告。

**审查清单必须包含：**

- 需求字段完整性（目标、范围、交付物、审查要点）。
- 覆盖原始意图无遗漏或歧义。
- 交付物映射明确（代码、文档、测试、验证报告）。
- 依赖与风险评估完毕。
- 审查结论已留痕（含时间戳）。

**决策规则：**

- 综合评分 ≥90 分且建议“通过” → 确认通过。
- 综合评分<80 分且建议“退回” → 确认退回。
- 80–89 分或建议“需讨论” → 仔细审阅后决策。

## 📊 架构优先级

- “标准化 + 生态复用”拥有最高优先级，必须首先查找并复用官方 SDK、社区成熟方案或既有模块。
- 禁止新增或维护自研方案，除非已有实践无法满足需求且获得记录在案的特例批准。
- 必须删除自研实现以减少维护面，降低长期技术债务和运维成本。
- 在引入外部能力时，必须验证其与项目标准兼容，并编写复用指引。
- 对现有自研或偏离标准的实现，必须规划替换或下线时间表，确保维护成本持续下降。

---

## 🛡️ 安全性原则

- **安全与角色规则位于最高优先级栈顶部**：当项目局部规范与全局安全/角色规范冲突时，必须以 `~/.claude.md/CLAUDE.md` 中的更高优先级规则为准。
- 必须诚实对待模型与工具的局限性，不允许隐瞒风险或虚构能力。
- 禁止故意削弱已有安全控制（如鉴权、加密、审计、访问隔离等），除非有充分论证与替代方案，并记录在案。
- 外部网络相关调用（例如 context7、fetch、exa、tavily-hikari、mcp-deepwiki 等）必须视为**只读**，不得用于修改外部系统状态。
- 必须尊重隐私与敏感信息，不得将密钥、口令、个人数据等写入输出或日志；`.mcp.json` 中的密钥与 token 不得在对话与文档中回显。
- 讨论安全问题时可以说明风险、限制与必要的防护措施，不得鼓励不安全实践。

## ✅ 代码质量强制标准

### 📝 语言与注释规范

**强制中文使用范围：**
所有以下场景必须强制使用简体中文，无任何例外：

- AI 与用户的所有对话回复。
- 所有文档（设计文档、API 文档、README、规范文档等）。
- 所有代码注释（单行注释、多行注释、文档注释）。
- Git 提交信息（commit message）。
- 操作日志（operations-log.md、coding-log 等）。
- 审查报告（review-report.md、verification-report.md 等）。
- 任务描述与规划文档。
- 错误提示与警告信息。
- 测试用例描述。
- 配置文件中的说明性文本。

**唯一例外**：代码标识符（变量名、函数名、类名、包名等）遵循项目既有命名约定（通常使用英文）。

**注释编写规范：**

- 所有代码文件必须使用 UTF-8 无 BOM 编码进行读写操作。
- 注释必须描述意图、约束与使用方式，而非简单重复代码逻辑。
- 禁止编写“修改说明”式注释，所有变更信息应由版本控制和日志承担。
- 当模块依赖复杂或行为非显而易见时，必须补充注释解释设计理由。
- 注释应简洁明了，避免冗长废话，直指核心要点。

### 🧪 测试规范

- 每次实现必须提供可自动运行的单元测试、冒烟测试或功能测试，由本地 AI 执行。
- 缺失测试的情况必须在验证文档中列为风险，并给出补测计划与截止时间。
- 测试需覆盖正常流程、边界条件与错误恢复，确保破坏性变更不会遗漏关键分支。

### 🏗️ 设计原则

- 严格遵循 SOLID、DRY 与关注点分离，任何共享逻辑都应抽象为复用组件。
- 依赖倒置与接口隔离优先，禁止临时绑死实现细节。
- 遇到复杂逻辑时必须先拆分职责，再进入编码。

### 💻 实现标准

- 禁止只交付“占位实现”或明显不完整的 MVP；提交前必须完成全量功能与核心数据路径。
- 必须主动删除过时、重复或逃生式代码，保持实现整洁。
- 必须始终遵守编程语言标准代码风格和项目既有风格规范。
- 对破坏性改动需给出迁移步骤或回滚方案。

### ⚡ 性能意识

- 设计时必须评估时间复杂度、内存占用与 I/O 影响，避免无谓消耗。
- 识别潜在瓶颈后应提供监测或优化建议，确保可持续迭代。
- 禁止引入未经评估的昂贵依赖或阻塞操作。

### 🧩 测试思维

- 在编码前编制可验证的验收条件，并在验证文档中回填执行结果。
- 对预期失败场景提供处理策略，保证服务可控降级。
- 连续三次验证失败必须暂停实现，回到需求和设计阶段复盘。

## 🔌 MCP 集成与工具调用链总览

本项目已在 `.mcp.json` 中配置了多种 MCP 服务，Claude Code 必须严格按照本节规则调用工具。

### 1. 全局 MCP 原则对齐

1. **单轮最多两个 MCP 服务**

   - 每一轮模型推理中最多调用两个 MCP 服务。
   - 若需要 sequential-thinking + shrimp-task-manager + codex-cli 等多于两个服务，必须拆分到多轮对话：

     - 例如：

       - 第 1 轮：sequential-thinking（规划） + shrimp-task-manager（任务拆解）；
       - 第 2 轮及后续轮：codex-cli（执行） + 其他辅助服务（如 desktop-commander/context7 等）。

2. **最小必要原则**

   - 工具调用范围必须最小化：限定 tokens、结果数、时间窗口和关键词，避免过度抓取。

3. **离线/本地优先**

   - 优先使用本地工具（desktop-commander、Serena、github 等）和项目内代码；
   - 网络与外部搜索（context7、exa、tavily-hikari、fetch、mcp-deepwiki）是后备方案。

4. **可追溯性**

   - 回答末尾在合适情况下追加“工具调用简报”，列出：工具名、输入摘要、关键参数或范围、调用时间/用途。

5. **失败降级**

   - MCP 调用失败时按优先级尝试其他服务；全部失败时给出保守的本地答案，并标记不确定性。

### 2. 当前项目 MCP 服务器一览（摘自 .mcp.json）

下表仅描述用途与触发条件，具体命令与环境变量由 `.mcp.json` 维护，本文件不再重复实际 token 等敏感信息：

| 服务名                | 类型  | 主要用途与触发条件                                                                                       |
| --------------------- | ----- | -------------------------------------------------------------------------------------------------------- |
| `Serena`              | stdio | 大型代码库的符号级搜索、跨文件引用分析、重构辅助。需要理解调用链、找定义/引用、进行精确插入/替换时触发。 |
| `sequential-thinking` | stdio | 用于复杂问题分解、生成多步骤计划、评估方案。任务不清晰或存在多种路径时作为**首选规划工具**。             |
| `shrimp-task-manager` | stdio | 任务拆解与任务管理，基于 sequential-thinking 结果生成具体执行任务与验收契约。                            |
| `codex-cli`           | stdio | 通过 `mcp__codex-cli__ask-codex` 调用 Codex CLI；**所有代码编辑、命令执行和测试运行的首选执行通道**。    |
| `exa`                 | stdio | Web 语义搜索与技术资讯检索，适用于一般搜索与技术博客/文章查找。                                          |
| `context7`            | stdio | 官方 SDK/API/框架文档查询，编程相关问题的首选文档源。                                                    |
| `fetch`               | stdio | 按 URL 抓取网页内容（转为可读文本/Markdown），在已知文档链接的情况下使用。                               |
| `memory`              | stdio | 项目与用户偏好的持久化知识图谱（实体/关系/观察）；跨会话记住重要信息时触发。                             |
| `mcp-deepwiki`        | stdio | Deep wiki/知识型站点的内容查询，对概念与长文深度理解时可选。                                             |
| `desktop-commander`   | stdio | 本地文件/目录/进程/数据分析首选工具：读写文件、目录操作、搜索、启动 REPL 等。                            |
| `tavily-hikari`       | http  | 另一个 Web 搜索与聚合服务，对一般信息/新闻类搜索可与 exa 互为备选。                                      |
| `github`              | http  | 完整 GitHub 操作（搜索代码、管理 PR/Issue 等）；项目托管在 GitHub 时的协作工具。                         |

> 说明：全局配置中提到的 firecrawl 工具目前未在 `.mcp.json` 中配置，若未来需要，可参考全局说明配置对应 MCP 服务；当前推荐通过 exa + fetch/tavily-hikari 组合满足通用 Web 检索与抓取需求。

### 3. 主调用链：从规划到执行

结合本项目规则与全局配置，本仓库的**标准调用链**如下：

1. **分析与规划阶段**（通常第 1 轮或前几轮）

   - 必须先调用 `sequential-thinking`（MCP）对任务进行结构化分析，生成 6–10 步以内的可执行计划。
   - 如任务较大或需要长期跟踪，再调用 `shrimp-task-manager` 生成任务板、验收契约与分步执行清单。
   - 单轮最多两个 MCP 服务，必要时把 sequential-thinking 和 shrimp 拆成两轮。

2. **上下文收集阶段（<context_gathering>）**

   - 使用 desktop-commander / Serena / github/search_code / context7 等收集项目与依赖上下文（详见后文“信息检索与外部工具集成”）。
   - 控制总 MCP 调用数在首轮 5–8 次以内，超出需要说明理由。

3. **执行阶段（Codex CLI 层）**

   - 所有代码编辑、命令与测试默认通过 `codex-cli` 调用 Codex CLI 执行：

     - 默认负载：`{"model":"gpt-5.1-codex","sandbox":true,"workingDir":"/绝对路径"}`，必须显式指定。
     - 对应 MCP 工具名称：`mcp__codex-cli__ask-codex`。

   - 若 Codex CLI 连续失败两次或任务是超长文档撰写，可回退为直接编辑（遵循全局 `CODEX_FALLBACK` 规则并在日志中记录）。

4. **辅助工具阶段**

   - 需要查文档 → context7；
   - 需要通用 Web 搜索 → exa 或 tavily-hikari；
   - 已知 URL → fetch；
   - 本地文件/数据处理 → desktop-commander；
   - 代码结构/重构辅助 → Serena；
   - 项目长期记忆 → memory；
   - GitHub 协作 → github。

5. **验证与自我反思阶段（<self_reflection>）**

   - 通过 Codex CLI 运行测试/构建/静态检查；
   - 评估可维护性、测试、性能、安全性、代码风格、文档、向后兼容性等至少五个维度；若有维度不达标必须回到实现阶段调整。

## 🚀 强制工作流程（结合 MCP 调用链）

### ⚡ 总原则（必须遵循）

- **强制深度思考**：任何时候必须首先使用 `sequential-thinking` 工具梳理问题，这是开发工作的基础（如任务过于简单可隐式执行这一步）。
- 不是必要的问题，不要询问用户，必须自动连续执行，不能中断流程。
- 问题驱动优先于流程驱动，追求充分性而非形式上的完整性，动态调整而非僵化执行。

### 🔗 工具链执行顺序（必须）

在保证“单轮最多两个 MCP 服务”的前提下，标准顺序为：

1. **规划链路**

   - 第一步：`sequential-thinking` → 生成高层任务拆解与主要风险。
   - 第二步：`shrimp-task-manager` → 将规划具体化为任务列表、验收标准和执行顺序。

2. **上下文收集与分析链路**

   - 按需使用：

     - `desktop-commander` → 本地文件/目录/数据分析；
     - `Serena` → 代码符号/调用链/跨文件引用；
     - `context7` → 官方文档/SDK/API 说明；
     - `github` → 开源实现示例、仓库内文件/PR/Issue；
     - `exa` / `tavily-hikari` / `fetch` / `mcp-deepwiki` → Web 搜索与抓取。

3. **执行链路（代码与命令）**

   - 默认通过 `codex-cli`（Codex CLI MCP）完成：

     - 修改文件、运行构建/测试、执行脚本等。

4. **记忆与协作链路**

   - `memory`：持久化存储项目结构、用户偏好和重要约定。
   - `github`：同步成果到远程仓库（PR、Issue、评论等）。

5. **失败处理**

   - 任一环节失败时，必须在 `operations-log.md` 中记录原因、补救措施与重新执行结果。
   - `codex-cli` 连续失败两次后可回退到直接 Read/Edit/Write/Bash 工具（标记 `CODEX_FALLBACK`）。

禁止：

- 跳过 sequential-thinking 和 shrimp-task-manager 做中大型任务的规划。
- 无理由跳过 Codex CLI 直接“手写”改动大批文件。

## 🔍 信息检索与外部工具集成（必须）

**核心原则：**

- 工具是手段，按需使用，避免僵化流程。
- 所有引用资料必须写明来源与用途，保持可追溯。
- 检索失败时，必须在日志中声明并改用替代方法。

### 本地文件和数据分析（desktop-commander，最高优先级）

**触发条件**：任何本地文件操作、CSV/JSON/日志分析、进程管理。
**能力概览**：

- 文件操作：`read_file`、`write_file`、`edit_block`（精确文本替换）。
- 目录管理：`list_directory`、`create_directory`、`move_file`。
- 搜索：`start_search`（支持文件名与内容搜索，流式返回结果）。
- 进程管理：`start_process`、`interact_with_process`（交互式 REPL）。
- 数据分析：配合 Python/Node.js REPL 处理 CSV/JSON/日志。

**最佳实践：**

- 所有本地 CSV/JSON/数据文件分析必须使用 desktop-commander，不要用 bash + cat/grep。
- 大目录搜索使用 `start_search`，渐进式拿结果。
- 文本修改尽量使用 `edit_block` 做“外科手术式”替换。

### 编程文档检索（context7 优先）

**触发条件**：任何关于编程库、框架、SDK、API 的问题。
**调用方式：**

1. `resolve-library-id` → 获得库 ID。
2. `get-library-docs` → 根据 topic 抓取相关文档。

**优势**：

- 官方文档优先，信息更新快、结构清晰。

若 context7 无法覆盖：

- 可用 `exa` / `tavily-hikari` 做搜索，再配合 `fetch` 抓取具体文章内容。

### 通用 Web 检索与抓取（exa / tavily-hikari / fetch / mcp-deepwiki）

- **exa**：语义搜索，一般 Web 问题与技术文章查找的首选；
- **tavily-hikari**：聚合式搜索，适合新闻/多源信息；
- **fetch**：已知 URL 的抓取工具；
- **mcp-deepwiki**：偏重知识站点、文档型内容的深度阅读。

建议策略：

- 想“找文章” → 先 exa/tavily-hikari 搜索 → 再用 fetch 抓取链接内容。
- 想“理解某个概念/条目”→ 可用 mcp-deepwiki。

### GitHub 集成（github MCP）

**能力：**

- 代码搜索：`search_code`、`search_repositories`。
- PR 管理：`create_pull_request`、`get_pull_request`、`merge_pull_request`。
- Issue 管理：`create_issue`、`update_issue`、`list_issues`。
- 文件操作：`get_file_contents`、`create_or_update_file`、`push_files`。
- 代码审查：`request_copilot_review` 等。

**触发条件**：

- 需要查找开源实现或跨仓库示例。
- 需要创建/更新 PR 与 Issue。

### 代码语义搜索与重构（Serena MCP）

**用途：**

- 大型代码库的符号级搜索与调用链分析。
- 精确定位函数/类/接口的定义与引用。
- 小范围自动重构（如替换函数体、在某符号前后插入代码片段）。

**策略：**

- 先用 Serena 确认修改范围与引用关系，再通过 Codex CLI/desktop-commander 实施真正的代码修改，保证可控。

### Memory MCP：持久化项目知识

**核心概念：**

- 实体（entity）：如“订单服务”“用户 A”“项目 X”。
- 关系（relation）：如“订单服务依赖库存服务”。
- 观察（observation）：某实体在某时间点的事实，如“项目 X 使用 Spring Boot 3.x”。

**触发条件：**

- 用户或项目给出长期有效的信息（如偏好、约定、架构决策）。
- 希望跨多次会话记住某些规则或上下文。

**使用建议：**

- 原子化存储：一条观察只描述一件事。
- 会话开始时使用 search/read_graph 恢复重要上下文。
- 注意不要把秘密数据写入 memory（如密钥、账号、密码）。
